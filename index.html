// 01. 백엔드 파트

#01 백엔드는 정확히 무엇을 하나요?

백엔드 = "주방"이야!
🍽️ 레스토랑에 비유하면:
프론트엔드 = 홀 직원 (손님 응대, 주문 받기, 음식 서빙)
백엔드 = 주방 (실제 음식 조리, 재료 관리, 레시피 보관)
DB = 냉장고 (재료들을 저장)

손님(사용자)이 "연차 신청 방법 알려줘!"라고 하면, 홀 직원(프론트엔드)이 주방(백엔드)에 주문을 넣고, 주방은 냉장고(DB)에서 재료 꺼내서 요리해서 다시 보내줘.
백엔드가 위드버디에서 구체적으로 하는 일
  • 🔐 로그인 처리 프론트에서 보낸 "회사코드 + 사번 + 이름"을 DB에서 찾아보고, 맞으면 accessToken 발급. 이게 "입장권"이야!
  • 💬 채팅 저장 사용자가 보낸 질문을 DB에 저장하고, AI 서버에 전달해서 답변을 받아 DB에 저장한 후 프론트에 돌려줘.
  • 📋 데이터 조회 채팅 기록, Nudge 목록, 빠른 질문 목록 등을 DB에서 꺼내서 프론트에 전달해줘.
  • 🛡️ 보안 검사 프론트가 보내는 요청에 올바른 토큰이 있는지 확인해. 없으면 401 에러를 돌려줘.
JWT 토큰이 뭐야? — 프론트엔드가 꼭 알아야 해!
JWT (JSON Web Token) = 로그인 성공 후 받는 "입장권"이야.

🎫 놀이공원에서 입장권 받으면 → 어떤 놀이기구든 보여주면 탈 수 있잖아?
JWT도 똑같아. 로그인 성공하면 긴 문자열을 받는데, 이후 모든 API 요청할 때 이걸 같이 보내면 돼.

어디에 저장? → localStorage 또는 메모리에 저장
어떻게 보내? → HTTP 요청 헤더에 Authorization: Bearer {토큰} 형태로 첨부
만료되면? → 401 에러 + TOKEN_EXPIRED 코드 → 로그인 페이지로 이동시켜야 해!
// 로그인 후 토큰 저장 예시 localStorage.setItem('accessToken', response.accessToken); // 이후 API 요청 시 토큰 첨부 fetch('/api/v1/chat/messages', { headers: { 'Authorization': `Bearer ${localStorage.getItem('accessToken')}`, 'Content-Type': 'application/json' } })
백엔드와 협업할 때 내가 확인해야 할 것들
  • ✅ 요청 URL이 정확한지 — 오타 하나도 에러남! 예: /api/v1/auth/login
  • ✅ 요청 Body 필드명 정확히 — companyCode와 company_code는 다른 거야!
  • ✅ HTTP 메서드 확인 — 조회는 GET, 생성은 POST. 틀리면 에러!
  • ✅ 에러 코드 처리 — 401이면 로그인 페이지로, 400이면 입력값 확인
  • ✅ Base URL 확인 — 개발 중엔 http://localhost:8080
// 02. AI 파트

#02 AI 파트는 무엇을 하나요?

RAG가 뭔데? — 위드버디 AI의 핵심!
RAG (Retrieval-Augmented Generation) = "문서를 참고해서 답변 만들기"

📚 예시로 이해하기:
일반 AI (GPT 같은 거): "아는 것만 말해요. 모르면 모른다고 해요."
RAG AI: "아, 이 회사의 복지카드 신청 문서를 먼저 읽어볼게요. 자, 문서 보니까 이렇게 하면 되네요!"

→ 위드버디는 각 회사의 사내 문서를 AI에게 먹여서, 그 회사에 맞는 답변을 만들어냄!
→ 그래서 A회사 신입이 물어보면 A회사 기준으로, B회사 신입이 물어보면 B회사 기준으로 대답해.
AI가 반환하는 답변 유형 3가지
rag_answer
문서 기반 답변 성공! 사내 문서에서 관련 내용을 찾아서 답해줬을 때. 가장 좋은 케이스야.
no_result
범위는 맞는데 문서가 없어! 예: "연차 신청법 알려줘" → 인사 관련이긴 한데, 이 회사 문서에 연차 내용이 없을 때.
out_of_scope
서비스 범위 밖! 예: "오늘 점심 메뉴 뭐가 좋아?" → 위드버디는 인사/행정 전용이라 이런 건 못 답해.
💡 프론트엔드에서 알아야 하는 이유: 백엔드에서 messageType을 그대로 돌려줘. 그래서 화면에서 messageType에 따라 다른 UI를 보여줘야 할 수도 있어! 예: no_result면 "관련 문서가 없어요" 안내 표시.
프론트엔드 ↔ AI 파트 협업 포인트
프론트엔드는 AI 서버에 직접 연락하지 않아!

흐름: 프론트 → 백엔드 → AI 서버 (내부 통신) → 백엔드 → 프론트

즉, 프론트 입장에선 백엔드에 질문을 보내면 답변이 돌아오는 것처럼 보여.
AI가 어떤 식으로 처리하는지는 신경 쓸 필요 없고, messageType과 content만 잘 받으면 돼!
⚠️ AI 파트와 맞춰야 할 것: 로딩 시간이 생각보다 길 수 있어. AI가 답변을 만드는 데 2~5초 걸릴 수 있으니, 프론트엔드에서는 반드시 "AI가 답변 중..." 로딩 표시를 보여줘야 해!

    백엔드가 만드는 API의 규격서. URL · 요청 방식 · 주고받는 데이터 형식이 전부 들어있어. 프론트엔드가 이걸 보고 fetch/axios 코드를 작성해.
  • 🗺️ ERD (데이터베이스 설계도)
    데이터가 어떤 구조로 저장되는지 보여주는 표. 직접 건드릴 일은 없지만, "어떤 데이터를 API로 받을 수 있는지" 파악하는 데 도움돼.
  • 🖼️ Figma 디자인 시안
    화면이 어떻게 생겨야 하는지 보여주는 설계도. 색상, 폰트 크기, 버튼 위치 등 모든 시각적 요소가 담겨 있어. 이걸 보고 HTML/CSS로 구현해!
Figma 읽는 법 — 초보자용
Figma는 디자이너가 사용하는 "디지털 설계도" 툴이야. 프론트엔드 개발자가 Figma를 보고 그대로 코드로 구현하는 거야.

보통 확인할 것들:
• 색상 코드 — 예: #2BCD9E (이걸 CSS에 그대로 쓰면 돼)
• 폰트 크기 — 예: 16px, 24px
• 간격 (Spacing) — 예: padding: 16px, margin: 24px
• 컴포넌트 구조 — 어떤 요소들이 어떻게 배치돼 있는지

Figma에서 수치 보는 법:
오른쪽 패널에 크기, 색상, 폰트 정보가 자동으로 나와. 요소 클릭하면 바로 확인 가능!
// 04. API 명세 읽기

#04 API 명세서 — 이렇게 읽어!

API가 뭐야? — 진짜 쉬운 비유
🍕 피자 주문 전화에 비유하면:

API = 피자집에 전화하는 방법
URL = 피자집 전화번호
요청 (Request) = 내가 하는 주문 ("페퍼로니 피자 1판 주세요")
응답 (Response) = 피자집이 주는 답변 ("네, 30분 뒤에 배달됩니다")

HTTP 메서드는 "어떤 종류의 요청인지" 알려주는 거야:
GET = 달라고 요청 (메뉴판 주세요)
POST = 새로 만들어 달라고 요청 (주문이요!)
PUT = 기존 것 수정해 달라고 요청 (치즈 추가해주세요)
DELETE = 삭제해 달라고 요청 (주문 취소요!)
🔐 인증 API
POST /api/v1/auth/login 로그인 ▼
사용자가 회사코드 + 사번 + 이름을 보내면, 서버가 확인하고 accessToken(입장권)을 발급해줘.

📤 요청 Body (프론트엔드가 보내는 데이터):

{ "companyCode": "1001", // 회사 식별 코드 "employeeNumber": "20260001", // 사번 "name": "김지원" // 이름 }

📥 응답 (200 OK — 로그인 성공):

{ "accessToken": "eyJhbGciOiJIUzI1NiIs...", // 이걸 저장해둬야 해! "user": { "id": 1, "companyName": "테크 주식회사", "name": "김지원", "hireDate": "2026-03-01" // 입사일 — Nudge 계산에 씀! } }

❌ 응답 (401 — 로그인 실패):

{ "code": "UNAUTHORIZED", "message": "회사코드, 사번 또는 이름이 올바르지 않습니다" }
💡 받은 accessToken은 localStorage에 저장하고, 이후 모든 API 요청 헤더에 Authorization: Bearer {토큰}으로 첨부해야 해!
💬 채팅 API
GET /api/v1/chat/messages 채팅 기록 조회 ▼
로그인한 사용자의 모든 채팅 기록을 시간 순서대로 불러와. 페이지 처음 들어왔을 때 호출!

📥 응답 (200 OK):

{ "messages": [ { "id": 101, "senderType": "USER", // 내가 보낸 메시지 "messageType": "user_question", "content": "복지카드는 어떻게 신청하나요?" }, { "id": 102, "senderType": "BOT", // AI가 보낸 메시지 "messageType": "rag_answer", // 문서 기반 답변 성공 "content": "복지카드는 관련 안내 문서를 기준으로..." } ] }
⚠️ senderType으로 말풍선 방향 결정! USER → 오른쪽, BOT → 왼쪽
POST /api/v1/chat/messages 질문 전송 ▼
사용자가 질문을 보내면, 백엔드가 AI에게 전달하고 답변까지 받아서 질문 + 답변 둘 다 돌려줘.

📤 요청 Body:

{ "content": "복지카드는 어떻게 신청하나요?" }

📥 응답 (201 Created):

{ "question": { // 내가 보낸 질문 (저장된 버전) "id": 201, "senderType": "USER", "content": "복지카드는 어떻게 신청하나요?" }, "answer": { // AI가 생성한 답변 "id": 202, "senderType": "BOT", "messageType": "rag_answer", // 또는 no_result, out_of_scope "content": "복지카드는 관련 안내 문서를..." } }
💡 이 API 응답을 받으면 question과 answer 둘 다 채팅 목록에 추가해줘야 해!
GET /api/v1/chat/quick-questions 빠른 질문 목록 ▼
채팅 화면 하단에 보여줄 추천 질문 버튼 목록. 버튼 클릭 시 그 content를 POST /chat/messages로 그냥 전송하면 돼!
{ "quickQuestions": [ { "content": "연차는 어떻게 신청하나요?" }, { "content": "급여일이 언제인가요?" }, { "content": "건강검진은 어떻게 받나요?" } ] }
👆 Nudge API
GET /api/v1/onboarding-suggestions/me 오늘의 Nudge 조회 ▼
오늘 날짜와 입사일을 비교해서 오늘 보여줄 Nudge 카드를 서버가 골라서 줘. 날짜 계산은 서버가 해!
{ "suggestions": [ { "id": 3, "title": "입사 1일차 안내", "content": "복지 제도와 사내 규정 문서를 먼저 확인해보세요.", "dayOffset": 1 // 입사 1일차에 보여주는 카드 } ] }
💡 빈 배열 []이 오면 오늘 해당하는 Nudge가 없는 거야. 이때 "오늘은 새로운 안내가 없어요" 같은 빈 화면 처리 해줘야 해!
// 05. ERD 읽기

#05 ERD — 데이터베이스 설계도 읽기

ERD가 뭐야?
ERD (Entity Relationship Diagram) = 데이터베이스 설계도

🗂️ 엑셀 스프레드시트를 상상해봐:
"사용자" 시트가 있고, "채팅 메시지" 시트가 있고, "회사" 시트가 있어.
ERD는 이 시트들이 어떤 컬럼으로 구성되고, 서로 어떻게 연결되는지 보여주는 거야.

프론트엔드가 ERD를 보는 이유: "API에서 어떤 데이터를 받을 수 있는지" 파악하기 위해!
companies — 회사 정보

로그인 시 회사 식별에 사용. company_code가 핵심!

컬럼타입키설명
idintPK회사 고유 번호
company_codevarchar회사 코드 (유일값, 로그인 시 사용)
namevarchar회사 이름
created_atdatetime생성일시
users — 사용자 정보

hire_date가 Nudge 계산의 핵심! 로그인 응답에서 받아와.

컬럼타입키설명
idbigintPK사용자 고유 번호
company_idbigintFK소속 회사 → companies.id 참조
namevarchar사용자 이름
employee_numbervarchar사번
hire_datedate입사일 (Nudge dayOffset 계산 기준!)
chat_messages — 채팅 메시지
